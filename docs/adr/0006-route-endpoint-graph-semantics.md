# ADR-0006: Route Endpoint Graph Semantics

Status: Proposed
Date: 2026-06-21

## Context

Metapi's route graph currently uses `entry` for two different jobs:

- downstream-visible ingress for public model names;
- internal route reuse for automatic groups, manual remaps, and macro outputs.

That dual meaning makes several behaviors hard to reason about:

- an `internal entry` is not actually an ingress, but it still looks like one;
- public duplicate checks must special-case generated macro and primitive
  entries;
- public/internal route group exposure is mixed with node visibility;
- manual macros choose source route ids even though the graph-level reusable
  thing is the route product generated from those routes;
- graph UI has to explain why a hidden or internal entry can still be selected
  as an upstream source.

The problem is conceptual, not just naming. A route graph needs three separate
semantic layers:

1. **Ingress**: where downstream requests enter the graph.
2. **Route product**: a reusable compiled route target that other graph objects
   can select.
3. **Execution target**: the concrete model/channel/token target eventually
   called.

The current `internal entry` collapses layers 1 and 2.

## Decision

Metapi will remove the internal-entry concept from the route graph model. The
graph will use `entry` only for downstream-visible ingress and will introduce a
first-class `route_endpoint` node for reusable route products.

The node vocabulary becomes:

```ts
type RouteGraphNode =
  | EntryNode
  | RouteEndpointNode
  | FilterNode
  | DispatcherNode
  | ModelEndpointNode
  | SyntheticEndpointNode
  | AutoNode;
```

`entry` is public-only:

```ts
type EntryNode = BaseRouteGraphNode & {
  type: 'entry';
  match: RouteGraphMatchSpec;
  selectionStrategy: RouteGraphSelectionStrategy;
};
```

`route_endpoint` represents an internally reusable route product:

```ts
type RouteEndpointNode = BaseRouteGraphNode & {
  type: 'route_endpoint';
  routeEndpointId: string;
  routeId?: number | null;
  exposure: 'public' | 'internal';
  ownerKind: 'automatic_route' | 'manual_route' | 'macro';
  sourceKind: 'route_group' | 'exact_model' | 'selector' | 'synthetic';
  backend: RouteGraphBackendSpec;
  match?: RouteGraphMatchSpec;
  metadata?: Record<string, unknown>;
};
```

The word `visibility` must no longer mean public exposure. Public/internal is a
route product **exposure** policy:

- `public`: generate an `entry` plus a `route_endpoint`;
- `internal`: generate only a `route_endpoint`.

Graph UI display concerns should remain separate. If the graph later needs a
display-only hidden state, that must be named separately, such as
`graphVisibility`, and must not affect downstream exposure.

## Core Semantics

### Entry

`entry` is an external ingress. It owns downstream model matching and is the
only graph node type that can expose a downstream model name.

Rules:

- every entry is public;
- entries must have a valid match/display name;
- public model names must be globally unique across entries;
- entries must reach an executable terminal path;
- entries are never used as an internal reusable route target.

Duplicate public model validation scans only `type === 'entry'`.

### Route Endpoint

`route_endpoint` is the reusable product generated by a route, route group, or
macro. It is selectable by manual macros and other route products.

Rules:

- route endpoints do not expose downstream model names by themselves;
- route endpoints do not participate in public duplicate validation;
- route endpoints must resolve to a compiled executable path;
- route endpoints can be public or internal by exposure, but both are reusable;
- internal route endpoints are valid even when no public entry reaches them.

Route endpoints are stable graph-level identities. Manual macros should select
route endpoints, not raw database route ids.

### Model Endpoint

`model_endpoint` remains the compiled/runtime executable primitive for concrete
upstream targets. It represents channels, tokens/API keys, upstream models, and
target-selection metadata after semantic route products have been lowered.

`model_endpoint` is not a semantic route product and is not the normal object a
manual route should reference. The semantic and catalog-facing endpoint
abstraction is `route_endpoint`; ADR-0007 refines that abstraction so concrete
upstream supplies and reusable route products are both represented as
`route_endpoint` with different endpoint kinds. A route endpoint may lower to
one or more model endpoints through generated selector/dispatcher logic.

## Macro Surface

Candidate selector macros will stop modeling external exposure as
`surface.entry.visibility`. The macro surface is split into ingress and output:

```ts
type CandidateSelectorMacroSurface = {
  ingress:
    | { kind: 'external'; match: RouteGraphMatchSpec }
    | { kind: 'embedded'; input: 'request' | 'bidirect' }
    | { kind: 'none' };
  output:
    | { kind: 'route_endpoint' }
    | { kind: 'bidirect' };
  ports: RouteGraphPort[];
};
```

Automatic route group macros:

- public exposure: `ingress.kind = 'external'`;
- internal exposure: `ingress.kind = 'none'`;
- both output a route endpoint.

Manual selector macros usually use external ingress and route-endpoint inputs.
Embedded macros can still receive bidirectional flow from another graph object.

## Macro Inputs

The `route_ids` macro input is replaced by route-endpoint references:

```ts
type CandidateSelectorGroupInput =
  | { kind: 'route_endpoints'; endpointIds: string[] }
  | { kind: 'endpoint_query'; cel: string }
  | { kind: 'model_pattern'; pattern: string }
  | { kind: 'inline_model_endpoints'; endpoints: ModelEndpointTarget[] }
  | { kind: 'synthetic'; statusCode: number; message: string };
```

Behavior:

- `route_endpoints` is the normal manual picker path and preserves user order;
- `endpoint_query` is the advanced dynamic path and receives endpoint metadata;
- `model_pattern` materializes route/model endpoint candidates by pattern;
- `inline_model_endpoints` is for explicitly declared upstream model targets;
- `synthetic` represents fallback/error responses.

Manual macros must not use database route ids as their primary graph input.

## Endpoint Catalog

The frontend needs a unified route endpoint catalog so manual macros can choose
automatic upstream route products without knowing about generated graph
internals.

The API exposes:

```http
GET /api/route-endpoints
```

Response items use a graph product vocabulary:

```ts
type RouteEndpointCatalogItem = {
  endpointId: string;
  routeId: number | null;
  label: string;
  exposure: 'public' | 'internal';
  ownerKind: 'automatic_route' | 'manual_route' | 'macro';
  sourceKind: 'route_group' | 'exact_model' | 'selector' | 'synthetic';
  enabled: boolean;
  displayIcon?: string | null;
  modelPattern?: string;
  publicModelName?: string | null;
  upstreamModels?: string[];
  siteNames?: string[];
  tags?: string[];
  metadata?: Record<string, unknown>;
};
```

The manual macro picker uses this catalog instead of `RouteSummaryRow[]` or
`routeIds[]`.

## Projection And Compilation

Automatic and list-created routes project as semantic route products first.

Automatic exact-model discovery has an additional grouping layer defined in
ADR-0007. A discovered upstream model target is not itself a public route
product. Automatic construction groups upstream model endpoints by canonical
public model and emits one stable automatic route group product endpoint for
manual reuse.

Public route:

```text
entry(public model)
  -> generated selector / dispatcher
  -> route_endpoint
  -> generated execution path
  -> model_endpoint | synthetic_endpoint
```

Internal route:

```text
route_endpoint
  -> generated selector / dispatcher
  -> generated execution path
  -> model_endpoint | synthetic_endpoint
```

The exact primitive layout may differ as long as the compiled graph preserves
these semantic contracts:

- an entry resolves to a route endpoint or executable path;
- a route endpoint resolves to executable path metadata;
- a manual macro can select route endpoints and dispatch through their compiled
  paths;
- compiled debug view can expand route endpoint internals without duplicating
  semantic macro nodes.

## List And Graph UX

The route list should be route-product centric:

- Public tab: products with `exposure = 'public'`;
- Internal tab: products with `exposure = 'internal'`;
- Manual tab: user-authored remaps/selectors.

Changing public/internal exposure regenerates ingress only:

- public -> internal: remove entry, keep route endpoint;
- internal -> public: add entry if the public model name is unique.

The graph default view shows semantic objects:

- public entries;
- macro nodes;
- route endpoint outputs when useful for reuse;
- no internal entries.

Compiled debug view may show generated dispatchers, cloned endpoints, and
read-only primitive edges. Semantic macro and its generated primitives must not
be shown as duplicate authoritative objects at the same time.

## Configuration Version

This is a breaking graph/config change. Metapi will bump the route graph
configuration version and will not preserve compatibility with internal-entry
graphs.

Migration rewrites existing saved graphs:

- delete every `entry` whose visibility was `internal`;
- create a route endpoint for each route projection/macro output;
- convert macro `route_ids` group inputs to `route_endpoints`;
- reconnect internal-entry consumers to the corresponding route endpoint;
- reject any remaining non-public entry during normalization/validation.

Imports using old internal-entry graphs must fail with a clear migration error
if they cannot be rewritten deterministically.

## Consequences

Benefits:

- public duplicate validation becomes simple and correct;
- public/internal route group behavior no longer depends on fake ingress nodes;
- manual macros select stable graph products instead of database route ids;
- automatic generated route groups can be reused cleanly by manual selectors;
- graph UI can separate semantic editing from compiled debug output.

Costs:

- route graph schema, compiler, runtime, API, frontend picker, and tests all
  need coordinated changes;
- existing route graph fixtures and snapshots must be rewritten;
- the endpoint catalog becomes a new shared contract between server and web;
- no-compat migration means old custom graph imports may require manual repair
  if automatic rewrite cannot identify route endpoint ownership.

## Non-Goals

- This ADR does not redesign model endpoint pricing or compatibility policy.
- This ADR does not require a new visual layout engine.
- This ADR does not make every generated primitive user-editable.
- This ADR does not remove model endpoints, synthetic endpoints, filters, or
  dispatchers.
