# 成本目录

成本目录负责维护 **参考价格数据库**。参考价格是一个对比基线，用于判断某个入口模型相对参考价更贵还是更便宜。它不代表 Metapi 调用上游时实际付出的成本。

运行时成本策略在 **系统设置 -> 成本策略** 中维护，包括上游默认价格、钱包估值、单位换算和价格漂移检查。两类配置的职责不同：

- 参考价格数据库：用于计算“相对参考价倍率”，没有命中时不计算参考倍率。
- 上游成本估算：用于理论成本、路由成本权重和仪表盘估值。
- 钱包估值：把站点余额、点数、credit 等换算成统一成本单位。
- 价格漂移检查：比较平台价格和实际计费样本，发现偏差后通过 Inbox 提醒。

## 价格解析关系

理论上游成本按以下顺序解析：

```text
模型级手动上游成本
  -> 上游平台目录价格
  -> 上游默认计价
```

参考倍率单独使用参考价格数据库：

```text
参考价格数据库命中 -> 计算参考倍率
未命中 -> 不显示参考倍率
```

参考价格和上游成本应保持分离。参考价格用于倍率对比；上游成本用于理论成本、路由权重和仪表盘估值。

## 参考价格数据库

入口：**成本目录 -> 参考价格数据库**

参考价格条目用于回答：这个模型相对于某个参考价格基线贵还是便宜。

每个条目包含：

- 模型名；
- provider；
- display name；
- aliases；
- 常规 token 价格；
- 高级 `PricingPlan` JSON；
- 来源 URL、来源类型和备注。

支持的维护方式：

- 在页面中新增或编辑条目；
- 从 JSON 文件导入；
- 粘贴 JSON 合并或替换导入；
- 配置远程 JSON 链接并按 Cron 同步；
- 手动立即同步。

远程同步的 JSON 可以是数组，也可以是包含 `entries` 字段的对象。

```json
{
  "entries": [
    {
      "id": "openai:gpt-5.5",
      "provider": "openai",
      "modelName": "gpt-5.5",
      "aliases": ["gpt-5.5-latest"],
      "simpleTokenPricing": {
        "inputPerMillion": 2,
        "outputPerMillion": 8
      },
      "sourceUrl": "https://example.com/pricing.json",
      "sourceType": "remote"
    }
  ]
}
```

复杂价格规则请使用高级计价方案，详见 [高级计价方案](./advanced-pricing.md)。

## 上游平台目录价格

部分 New API / One API / OneHub 风格站点会返回模型目录价格。Metapi 会把它作为“平台目录价格”参与上游成本估算。

平台目录价格的定位：

- 它是上游报告的候选价格；
- 它不是参考价格数据库；
- 它可能受到分组倍率、站点策略或上游实现差异影响；
- 没有手动上游成本时，系统可以用它估算理论成本。

如果上游平台没有价格接口，或返回数据不完整，系统会继续使用上游默认计价。

## 手动上游成本

手动上游成本在具体站点、账号、Token 或模型入口维护。它用于覆盖平台目录价格。

适用场景：

- 上游不是 New API，无法自动返回价格；
- 平台目录价格不可信；
- 你有自己的合同价；
- 某个账号或 API Key 使用不同折扣。

手动上游成本只代表“调用这个上游供应的成本”，不是下游计费价格。

## 上游默认计价

入口：**系统设置 -> 成本策略**

上游默认计价是最后一级默认成本来源。它用于：

- 理论成本估算；
- 路由成本权重；
- 没有手动成本和平台目录价时避免成本为空。

它不用于参考倍率。参考倍率没有数据库命中时应该留空。

## 钱包估值

很多上游站点的余额不是 USD，而是 CNY、credit、点数、quota 或平台自定义单位。钱包估值用于把这些余额折算成系统统一成本单位。

估值链路：

```text
上游平台扣费单位
  -> 钱包单位成本
  -> 充值折扣
  -> 单位换算
  -> 系统基准成本单位
```

可配置层级：

- 站点；
- 账号；
- Token。

优先级：

```text
Token 钱包成本
  -> 账号钱包成本
  -> 站点钱包成本
  -> 系统默认余额估值
```

继承模式：

| 模式 | 含义 |
|------|------|
| 继承 | 当前层级不覆盖，继续使用上级配置 |
| 覆盖 | 当前层级使用自己的钱包成本 |
| 禁用 | 当前层级明确不使用钱包估值 |

钱包成本字段：

| 字段 | 说明 |
|------|------|
| 钱包单位 | 上游余额或点数单位，例如 `USD`、`CNY`、`credit`、`sitea-credit` |
| 单位成本倍率 | 1 个钱包单位按多少现金成本计算 |
| 充值折扣 | 实际购买成本倍率，`1` 表示原价，`0.8` 表示八折 |
| 每日获得余额 | 签到或任务每天稳定获得的余额 |
| 每日来源 | 手动、签到观测、混合或无 |
| 可信度 | 精确、估算或数据不完整 |

对于免费签到额度，系统不强行换算成现金。它会单独展示免费额度消耗，例如“本次请求消耗 0.2 天签到额度”。

## 单位换算

入口：**系统设置 -> 成本策略 -> 单位换算**

单位换算把不同钱包单位折算到系统基准成本单位。常见例子：

```text
CNY -> USD
sitea-credit -> USD
quota -> USD
```

规则：

- `USD -> USD` 这类同单位条目不需要配置，系统按 1:1 处理；
- 重复单位对会被拒绝；
- 方向会自动归一化，系统会保存为稳定方向并自动调整换算率；
- 成本计算使用最新有效记录；
- 历史日志应该保留当时使用的快照，不应随新汇率漂移。

## 价格漂移检查

入口：**系统设置 -> 成本策略 -> 价格漂移检查**

价格漂移检查比较：

- 上游平台目录价格；
- 实际计费或代理日志样本；
- 当前配置的容差。

当样本数达到门槛，且偏差超过相对或绝对容差时，系统会产生需要关注的事项。

可配置项：

| 配置 | 说明 |
|------|------|
| 窗口小时 | 检查最近多少小时的数据 |
| 最小样本数 | 样本不足时不判断漂移 |
| 相对容差 | 百分比偏差门槛 |
| 绝对容差 | 金额偏差门槛 |

漂移检查属于平台价格诊断，不属于参考价格数据库。

## 仪表盘估值

仪表盘中的站点分布、模型分析和余额相关图表应使用估值后的金额，而不是把不同站点的原始 point 简单相加。

如果某些站点缺少钱包成本或单位换算，图表会显示估值覆盖情况。覆盖不完整时，应优先补齐：

1. 站点或账号的钱包成本；
2. 非基准单位的换算率；
3. 上游默认计价；
4. 需要时的模型级手动成本。

## 建议配置顺序

1. 在系统设置里确认基准成本单位。
2. 配置常用单位换算。
3. 为站点、账号或 Token 配置钱包成本。
4. 保持上游平台价格默认启用。
5. 对没有自动价格的模型配置手动上游成本。
6. 在成本目录维护参考价格数据库。
7. 启用价格漂移检查。
